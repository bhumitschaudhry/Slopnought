# Anti-Patterns: How LLM-Generated Code Fails

This is a working catalog of the specific ways code generated by language models — including Claude — tends to degrade over time, even when it's correct at the moment it's written. Both code generation mode and refactoring mode exist largely to counteract these patterns. Read this once per session and keep it active in mind; in refactoring mode, this list doubles as a literal scan checklist.

The common thread: most of these come from optimizing for *this turn* rather than for *the codebase as a whole over time*. An LLM agent, by default, tends to treat each task as a fresh, isolated problem. Maintainable software requires treating each task as one increment in a continuous, shared, long-lived system.

## 1. Duplicated logic

**What it looks like:** The same validation rule, calculation, formatting logic, or business rule appears in multiple places with slightly different wording or implementation, because it was easier to write it again than to find and reuse the existing version.

**Why it happens:** Searching the codebase for an existing implementation takes more effort in the moment than writing a new one. Without that search, duplication is invisible until two copies drift apart and start disagreeing.

**Prevention:** Before implementing any non-trivial logic — especially validation, calculations, formatting, or anything that encodes a business rule — search the codebase for something that already does this or something close to it. If found, reuse or extend it. If a near-duplicate is found, that's a signal to extract a shared abstraction, not to add a third copy.

## 2. Context collapse

**What it looks like:** Code that made sense in the narrow context of the immediate request but ignores or contradicts the broader system it's being inserted into — e.g., a new endpoint that bypasses the auth middleware every other endpoint uses, or a new data model that doesn't follow the schema conventions the rest of the database follows.

**Why it happens:** The agent solves the stated problem correctly in isolation but doesn't load enough surrounding context to notice the solution doesn't fit its environment.

**Prevention:** Before writing code, look at how the surrounding system already solves similar problems — similar files, similar layers, similar entry points. New code should look like it grew from the same codebase, not like it was pasted in from a different project.

## 3. Brittle shortcuts

**What it looks like:** Hardcoded values that should be config, magic numbers without explanation, string-matching where an enum or typed structure belongs, sleep() calls papering over race conditions, catching and silently swallowing exceptions to make an error go away, disabling a test instead of fixing it.

**Why it happens:** Shortcuts make the immediate task "work" fastest. They're invisible in a demo and only surface as production incidents or confusing failures later, often far from where the shortcut was taken.

**Prevention:** When a shortcut is taken (and sometimes a shortcut is the right call under real constraints), say so explicitly — in a comment and ideally to the user — rather than letting it pass as if it were a considered, permanent design. A shortcut without a flag is a trap for the next person.

## 4. Inconsistent conventions

**What it looks like:** Mixed naming styles (camelCase next to snake_case in the same language), inconsistent error handling strategies (some functions throw, some return null, some return error objects), inconsistent file/folder organization, formatting that doesn't match the linter config already in the project.

**Why it happens:** Each piece of code is generated somewhat independently, without cross-checking against the conventions established elsewhere in the same codebase.

**Prevention:** Before writing in an existing codebase, identify its conventions (naming, error handling, module layout, testing style) and match them, even if a different convention is "more correct" in the abstract. Raise the inconsistency to the user if it's bad enough to warrant a deliberate, codebase-wide fix — don't unilaterally introduce a competing convention.

## 5. Hidden dependencies

**What it looks like:** A function that works only because it's called in a specific order relative to other functions, global mutable state that several unrelated modules quietly depend on, implicit coupling between a frontend and backend that isn't expressed in any shared type or contract, configuration that one module assumes another has already set up.

**Why it happens:** Implicit dependencies are easy to introduce and invisible until something is reordered, parallelized, or extracted — at which point they fail in confusing, non-local ways.

**Prevention:** Dependencies should be explicit — function parameters, constructor injection, documented preconditions — rather than implied by execution order or shared mutable state. If a hidden dependency already exists in the code being touched, surface it in a comment or doc rather than silently relying on it.

## 6. Missing rationale

**What it looks like:** Code that does something unusual, non-obvious, or counter to what a reader would expect, with no comment explaining why. Six months later, nobody knows if the unusual thing is load-bearing (don't touch it) or vestigial (safe to delete), so it never gets touched and never gets cleaned up.

**Why it happens:** The reason for a decision is obvious in the moment it's made and therefore feels unnecessary to write down. It stops being obvious the instant that context window closes.

**Prevention:** Any time code deviates from the obvious/default approach, write down why, right next to the code. "Why" comments age far better than "what" comments — the what is recoverable by reading the code; the why is only recoverable if someone wrote it down.

## 7. Weak validation

**What it looks like:** Trusting input shapes without checking them, assuming an API response always has the expected fields, skipping bounds/null/type checks because "it should never happen," validation that checks format but not meaning (e.g., confirms a string is non-empty but not that it's a plausible value).

**Why it happens:** Happy-path code is faster to write and passes the same tests that prompted it. Validation gaps don't surface until real-world, messy input arrives — which is exactly when they're most costly to debug.

**Prevention:** Validate at every trust boundary (user input, external API responses, file/database reads, function arguments in public interfaces). Fail loudly and specifically rather than silently proceeding with bad data or throwing a generic error far from the actual problem.

## 8. Architecture drift

**What it looks like:** A codebase that started with a clear layering (e.g., controllers → services → repositories) gradually accumulates code that skips layers, business logic that leaks into UI components, a "utils" file that becomes an unstructured dumping ground, new features each inventing their own slightly-different pattern instead of following the established one.

**Why it happens:** Each individual change seems small and locally reasonable; the cumulative effect of many such changes is a codebase whose actual structure no longer matches its intended structure, and nobody decided that on purpose.

**Prevention:** Before adding code, identify which architectural layer/boundary it belongs to and put it there, even if a shortcut elsewhere would be faster for this one task. If an existing architecture is being violated repeatedly because it no longer fits the system's needs, that's a signal for a deliberate, documented change to the architecture — not an invitation to ignore it task-by-task.

## How these connect

These aren't independent — they compound. A brittle shortcut (#3) taken without rationale (#6) becomes a hidden dependency (#5) that the next change duplicates instead of reuses (#1) because nobody could tell it was meant to be reused, in a part of the codebase that's already drifted from its intended architecture (#8) because the conventions were inconsistent (#4) and nothing validated that the drift was happening (#7).

The fix for all of them is the same discipline: **look before you write** (search for existing patterns/logic), **write for the reader, not just the compiler** (names, comments, structure), and **say the quiet part out loud** (rationale, shortcuts, tradeoffs — in comments, docs, or directly to the user).
